2. Create `src/lib/rate-limit.ts`:
   - Import `Ratelimit` from `@upstash/ratelimit`, `Redis` from `@upstash/redis`
   - Create shared Redis client: `Redis.fromEnv()`
   - Create and export TWO rate limiter singletons OUTSIDE handler for edge caching:
     - `hourlyLimit`: `Ratelimit({ redis, limiter: Ratelimit.slidingWindow(20, "1 h"), analytics: true, prefix: "@upwork-ai-chatbot:hourly" })`
     - `dailyLimit`: `Ratelimit({ redis, limiter: Ratelimit.slidingWindow(100, "1 d"), analytics: true, prefix: "@upwork-ai-chatbot:daily" })`
   - Export `checkRateLimit(identifier: string)` function that:
     - Calls BOTH `hourlyLimit.limit(identifier)` and `dailyLimit.limit(identifier)`
     - Returns the TIGHTER limit (whichever has lower remaining count)
     - Return type: `{ success: boolean, limit: number, remaining: number, reset: number }`
     - Logic: `success = hourlyResult.success && dailyResult.success`, `remaining = Math.min(hourlyResult.remaining, dailyResult.remaining)`, use reset from whichever limit is tighter
   - Export `getIpAddress(request: NextRequest): string` utility:
     - Try `request.ip` first (Vercel provides this)
     - Fallback to `x-forwarded-for` header (first IP in comma-separated list)
     - Last resort: "127.0.0.1"

3. Create `src/lib/cost-tracking.ts`:
   - Import `Redis` from `@upstash/redis`
   - Create Redis client via `Redis.fromEnv()`
   - Define PRICING object for gpt-4.1-mini:
     - input: $0.15 / 1,000,000 tokens
     - output: $0.60 / 1,000,000 tokens
   - Define DAILY_BUDGET = 10 (dollars)
   - Define ALERT_THRESHOLDS: warning (50%, $5), critical (80%, $8), shutoff (100%, $10)
   - Export `getCurrentCost(): Promise<number>`:
     - Get today's date string (YYYY-MM-DD)
     - Read `cost:{date}` key from Redis
     - Return 0 if not found or different day
   - Export `trackChatCost(inputTokens: number, outputTokens: number): Promise<void>`:
     - Calculate cost from tokens * pricing
     - Atomically increment Redis `cost:{date}` key (use `redis.incrbyfloat`)
     - Set TTL to 172800 (2 days) if key is new
   - Export `checkBudgetRemaining(): Promise<{ allowed: boolean, current: number, budget: number, percentUsed: number }>`:
     - Get current cost, calculate percent, return allowed status
   - Export `checkCostAlerts(currentCost: number): { level: 'ok' | 'warning' | 'critical' | 'shutoff', message: string }`:
     - Compare against thresholds, return appropriate level and message
  </action>

       - Call checkRateLimit(ip)

   - Update matcher to include both admin and API routes:
     ```
     matcher: ["/admin/:path*", "/api/chat"]
     ```

   IMPORTANT: Rate limiting should only fire on `/api/chat` POST requests (the actual message sends), NOT on page loads like `/chat` or widget page loads. This prevents burning rate limit on page navigations.

   WIDGET VERIFICATION: The embedded widget's chat messages are sent to the SAME `/api/chat` endpoint via its iframe. Confirm this by reading `src/components/widget/WidgetEmbed.tsx` or the widget page to verify it posts to `/api/chat`. Since middleware intercepts `/api/chat` regardless of origin (full-page or widget iframe), both share the same per-IP rate limit counter. No separate widget rate limiting is needed.
